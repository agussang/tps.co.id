BAB I
PENDAHULUAN

1.1 Latar Belakang

Sebagai bagian dari proyek pengembangan sistem CMS (Content Management System) website PT Terminal Petikemas Surabaya, telah dilakukan instalasi dan konfigurasi application server pada infrastruktur jaringan internal TPS. Application server berfungsi sebagai host dari aplikasi CMS yang melayani website publik dan panel admin untuk pengelolaan konten.

Dalam proses implementasi, ditemukan beberapa kendala teknis yang memerlukan penanganan khusus, terutama terkait dengan keterbatasan akses internet pada server. Kendala ini berdampak signifikan terhadap proses deployment, pemeliharaan, dan pengembangan berkelanjutan dari sistem CMS.

Dokumen ini memaparkan secara lengkap proses instalasi, kendala yang dihadapi beserta solusinya, serta analisis mendalam mengenai mengapa akses internet sangat diperlukan untuk mengakomodir proses deployment dan pemeliharaan server secara optimal.

1.2 Tujuan Dokumen

Dokumen ini disusun dengan tujuan:

  1. Mendokumentasikan proses instalasi dan konfigurasi application server (172.19.154.93).
  2. Mendokumentasikan kendala teknis yang ditemui selama proses implementasi beserta solusinya.
  3. Menjelaskan sistem deployment yang saat ini diterapkan dan keterbatasannya.
  4. Memberikan analisis dan justifikasi teknis mengapa akses internet sangat dibutuhkan untuk menunjang proses deployment, pemeliharaan, dan pengembangan server.
  5. Memberikan rekomendasi konfigurasi akses internet yang aman.

1.3 Ruang Lingkup

Ruang lingkup dokumen ini meliputi:

  1. Instalasi Docker Engine dan konfigurasi container aplikasi pada server 172.19.154.93.
  2. Konfigurasi volume mount untuk deployment system.
  3. Dokumentasi seluruh kendala teknis dan bug fixing yang dilakukan.
  4. Analisis komprehensif kebutuhan akses internet untuk server.

# Verifikasi instalasi

# Enable Docker service

4.1 Prisma Client Initialization Failure

SeverityCRITICAL — Seluruh halaman website tidak dapat diakses
Tanggal Ditemukan12 Februari 2026
File Terkaitpkgs/utils/prisma.ts

Deskripsi Masalah

Seluruh halaman website (homepage, berita, unduh dokumen, dll) menampilkan error atau halaman kosong. Log container menunjukkan error berulang: TypeError: undefined is not an object (evaluating 'db.config').

Root Cause Analysis

Prisma ORM versi 5.11.0 memiliki bug pada schema parser (psl/parser-database/src/walkers/relation/inline.rs) yang menyebabkan perintah prisma generate gagal dengan error "index out of bounds: the len is 0 but the index is 0".

Dalam kode original, perintah prisma generate dan import PrismaClient berada dalam satu blok try-catch yang sama. Ketika prisma generate gagal, seluruh blok di-skip — termasuk import PrismaClient yang sebenarnya masih bisa berfungsi menggunakan client yang sudah di-generate sebelumnya saat Docker build. 

# Kode sebelum perbaikan (BERMASALAH)
try {
  await prisma generate;      // ← GAGAL (bug Prisma 5.11.0)
  g.db = new PrismaClient();  // ← TIDAK PERNAH DIEKSEKUSI
} catch (e) {
  // g.db tetap undefined → seluruh website error
}

Solusi

Memisahkan prisma generate dan import PrismaClient ke dalam dua blok try-catch independen, sehingga kegagalan generate tidak menghalangi penggunaan client yang sudah ada. 

# Kode setelah perbaikan
try {
  await prisma generate;
} catch (e) {
  console.log("Using existing client");  // Non-fatal
}

try {
  g.db = new PrismaClient();  // ← Tetap dieksekusi
} catch (e) {
  console.log("Prisma not initialized");
}

4.2 File Upload Tidak Berfungsi

SeverityHIGH — Admin tidak dapat upload dokumen/media
Tanggal Ditemukan12 Februari 2026
File Terkaitapp/srv/api/_tpsadmin_edit.ts, pkgs/api/_upload.ts

Deskripsi Masalah

Fitur upload file pada halaman admin edit content (/backend/tpsadmin/edit/:id) tidak berfungsi. Terdapat tiga bug yang saling terkait:

Bug 1: Request Body Format Salah

Frontend mengirim raw File object sebagai request body, padahal backend menggunakan multipart parser yang membutuhkan FormData dengan boundary. 

# Sebelum (SALAH)
fetch('/_upload', { method: 'POST', body: file })

# Sesudah (BENAR)
const formData = new FormData();
formData.append('file', file);
fetch('/_upload', { method: 'POST', body: formData })

Bug 2: Direktori Upload Tidak Dibuat

Backend menggunakan path relatif untuk mengecek dan membuat direktori upload, serta memiliki logika yang terbalik (hanya membuat direktori jika sudah ada). 

# Sebelum (SALAH — relative path + logika terbalik)
if (await existsAsync(dirname(to))) {
  dirAsync(dirname(to));  // No-op: sudah ada
}

# Sesudah (BENAR — absolute path, selalu buat)
const fullPath = dir(`${g.datadir}/files/${to}`);
await dirAsync(dirname(fullPath));

Bug 3: Page Reload Sebelum Save

Setelah upload berhasil, halaman langsung melakukan location.reload() tanpa menyimpan file path ke database terlebih dahulu, sehingga path yang baru diupload hilang.

Solusi: Mengganti location.reload() dengan update preview langsung di DOM, sehingga user bisa melihat file yang diupload dan menyimpannya melalui tombol "Simpan".

4.3 Routing File URL Tidak Konsisten

SeverityMEDIUM — File download dari halaman unduh dokumen error 404
File Terkaitpkgs/server/create.ts

Deskripsi Masalah

Halaman Prasi legacy (unduh-dokumen) menghasilkan URL file dengan pola /file/..., sedangkan endpoint server hanya menangani /_file/... (dengan underscore). Akibatnya, klik download file menghasilkan error 404 Not Found.

Solusi

Menambahkan redirect permanen (HTTP 301) dari /file/* ke /_file/* pada server routing.

4.4 Encoding Spasi pada URL File Legacy

SeverityMEDIUM — File dengan spasi di nama tidak bisa didownload
File Terkaitpkgs/server/create.ts

Deskripsi Masalah

Prasi legacy mengencode karakter spasi sebagai tilde (~) pada URL file, sedangkan file di disk menggunakan spasi. Contoh: URL /file/media/laporan~penggantian~ssl.pdf mengarah ke file laporan penggantian ssl.pdf.

Solusi

Menambahkan konversi ~ ke %20 (spasi) pada redirect handler.

5.2 Deploy Script (deploy.sh)

Script deployment otomatis telah dibuat untuk menyederhanakan proses. Script ini melakukan sinkronisasi 4 direktori source code, restart container, dan verifikasi health check. 

#!/bin/bash
# Usage:
#   ./deploy.sh              # Sync + restart
#   ./deploy.sh --sync-only  # Sync + clear cache
#   ./deploy.sh --restart    # Restart only

# 1. Sync source code via rsync (SSH)
rsync -az pkgs/     root@172.19.154.93:/opt/tps-app/pkgs/
rsync -az app/srv/  root@172.19.154.93:/opt/tps-app/app/srv/
rsync -az app/db/   root@172.19.154.93:/opt/tps-app/app/db/
rsync -az app/web/deploy/ root@172.19.154.93:/opt/tps-app/app/web/deploy/

# 2. Restart container
ssh root@172.19.154.93 "docker restart tps-app"

# 3. Health check & smoke test
# → Verifikasi container Up (healthy) dan homepage HTTP 200

6.4 Kebutuhan Internet untuk Package Management

Aplikasi CMS bergantung pada 247 npm packages yang dikelola oleh Bun package manager. Setiap package dapat memiliki security vulnerability yang ditemukan setelah deployment. Tanpa akses internet, proses update dan patching library tidak dapat dilakukan langsung pada server.

Skenario Risiko

Jika ditemukan critical vulnerability (contoh: Remote Code Execution) pada salah satu dari 247 dependencies, proses patching tanpa internet memerlukan:

  1. Developer update package di PC lokal.
  2. Rebuild seluruh Docker image di PC lokal.
  3. Transfer Docker image (1.17 GB) ke server melalui scp.
  4. Load image dan restart container di server.

Proses ini memakan waktu 30-60 menit dibandingkan 5 menit jika server memiliki akses internet (cukup bun install + restart).

6.5 Kebutuhan Internet untuk Security Updates

6.6 Kebutuhan Internet untuk Docker Registry

Docker image perlu diperbarui secara berkala — baik untuk update base image (security patches) maupun untuk perubahan dependencies. Tanpa akses ke Docker Hub (hub.docker.com) dan npm registry (registry.npmjs.org), proses rebuild Docker image tidak dapat dilakukan di server.

Proses Rebuild Saat Ini (Tanpa Internet)

  1. Build Docker image di PC developer lokal.
  2. docker save image ke file tar (~1.2 GB).
  3. scp file tar ke server (tergantung bandwidth internal).
  4. docker load image di server.
  5. Restart container dengan image baru.

Proses Rebuild Ideal (Dengan Internet)

  1. SSH ke server.
  2. docker compose build && docker compose up -d (otomatis download semua yang dibutuhkan).

6.7 Kebutuhan Internet untuk Monitoring dan Alerting

Sistem produksi memerlukan monitoring untuk mendeteksi masalah secara proaktif. Tanpa akses internet, notifikasi error tidak dapat dikirimkan melalui email, Slack, atau layanan monitoring seperti UptimeRobot, Datadog, atau Sentry.

Saat ini, jika server mengalami masalah di luar jam kerja, tidak ada mekanisme notifikasi otomatis. Masalah baru diketahui ketika ada laporan dari pengguna.

6.9 Rekomendasi Konfigurasi Akses Internet

Untuk menjaga keamanan jaringan internal TPS sambil memenuhi kebutuhan operasional server, direkomendasikan konfigurasi akses internet yang terbatas (whitelisted) — hanya mengizinkan koneksi ke layanan-layanan yang dibutuhkan.

Opsi A: Whitelist Domain (Recommended)

Mengizinkan server mengakses hanya domain-domain tertentu melalui firewall atau proxy:

Tabel 6.5 — Daftar Domain yang Dibutuhkan

Opsi B: VPN atau Proxy Terbatas

Menggunakan forward proxy (seperti Squid) yang hanya mengizinkan koneksi ke domain yang di-whitelist. Semua traffic internet dari server melalui proxy, sehingga IT Security TPS dapat memonitor dan mengontrol traffic.

Opsi C: Scheduled Internet Access

Memberikan akses internet hanya pada waktu tertentu (contoh: jam maintenance 00:00-06:00) untuk keperluan update dan deployment, dan menutup akses di luar jam tersebut.

Rekomendasi: Opsi A (Whitelist Domain) memberikan keseimbangan terbaik antara keamanan dan operasional. Server hanya bisa mengakses layanan yang benar-benar dibutuhkan, sementara IT Security tetap memiliki kontrol penuh.
