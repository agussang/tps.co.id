BAB I
PENDAHULUAN

1.1 Latar Belakang

Pada laporan sebelumnya (TPS-PRIMALINK/APP/2026-002, tertanggal 12 Februari 2026), telah didokumentasikan proses instalasi application server beserta kendala yang dihadapi. Salah satu kendala utama yang ditemukan pada saat itu adalah tidak tersedianya akses internet pada server (172.19.154.93), sehingga proses deployment harus dilakukan secara manual menggunakan rsync melalui SSH dari komputer developer.

Berdasarkan pengujian yang dilakukan pada tanggal 6 Maret 2026, ditemukan bahwa server telah memiliki akses internet penuh. Perubahan kondisi ini membuka peluang untuk menyederhanakan proses deployment menggunakan Git secara langsung di server, menggantikan mekanisme rsync manual yang sebelumnya digunakan.

Selain perubahan metode deployment, dokumen ini juga mendokumentasikan perubahan pada file docker-compose.yml agar volume mount mengarah langsung ke path Git repository. Pendekatan ini menghilangkan kebutuhan symbolic link (symlink) yang sebelumnya digunakan, sehingga proses recovery server menjadi lebih cepat dan sederhana.

1.2 Tujuan Dokumen

Dokumen ini disusun dengan tujuan:

  1. Mendokumentasikan hasil pengujian konektivitas server secara menyeluruh (internet, LAN, Docker, dan layanan website).
  2. Menjelaskan perubahan sistem deployment dari rsync manual menjadi git pull langsung di server.
  3. Mendokumentasikan perubahan konfigurasi docker-compose.yml yang disesuaikan dengan struktur direktori Git repository.
  4. Menyediakan prosedur recovery server yang cepat dan jelas untuk skenario disaster recovery.
  5. Menyediakan panduan operasional yang jelas dan lengkap untuk pengelolaan server ke depan.

1.3 Ruang Lingkup

Ruang lingkup dokumen ini meliputi:

  1. Pengujian konektivitas server (20 test case).
  2. Konfigurasi Git repository di server untuk deployment.
  3. Perubahan docker-compose.yml dan struktur volume mount.
  4. Prosedur recovery server dari nol (disaster recovery).
  5. Panduan deployment, maintenance, dan troubleshooting.

BAB III
PERUBAHAN SISTEM DEPLOYMENT

3.1 Perbandingan Workflow: Sebelum dan Sesudah

Dengan tersedianya akses internet, sistem deployment telah diubah dari mekanisme rsync manual menjadi git pull langsung di server. Selain itu, konfigurasi docker-compose.yml diubah agar volume mount mengarah langsung ke path di dalam Git repository, tanpa memerlukan symbolic link (symlink). Pendekatan ini mempercepat proses recovery server karena cukup git clone + docker compose up tanpa langkah tambahan. 

 SEBELUMNYA (rsync + symlink)                SEKARANG (git pull, tanpa symlink)
 ====================================        ====================================

 Developer PC        Server                  Developer PC        Server
 ┌──────────┐  SSH   ┌──────────┐            ┌──────────┐        ┌──────────┐
 │ git repo │──────▶ │ /opt/    │            │ git repo │        │ git repo │
 │          │ rsync  │ tps-app/ │            │          │        │ /opt/    │
 │          │ manual │ + symlink│            │          │        │ tps-app/ │
 └──────────┘        └──────────┘            └──────────┘        └──────────┘
      │                                           │  git push         ▲ git pull
      │ git push                                  ▼                   │
 ┌──────────┐                                ┌──────────┐────────────┘
 │  GitHub  │ ← Tidak bisa                   │  GitHub  │  Server akses
 │          │   diakses server               │          │  langsung
 └──────────┘                                └──────────┘

3.2 Setup Git Repository di Server

Git repository telah di-clone langsung ke server dari GitHub. Berikut adalah langkah-langkah yang telah dilakukan dan dapat diulangi jika diperlukan setup ulang.

1 Login ke Server

Login ke server melalui AnyDesk (remote desktop) atau SSH:

ssh root@172.19.154.93

Anda berhasil login jika tampilan berubah menjadi: root@tpswebsiteapp:~#

2 Clone Repository dari GitHub

Ketik perintah berikut lalu tekan Enter. Proses clone akan memakan waktu beberapa menit tergantung kecepatan internet.

cd /opt
git clone https://github.com/agussang/tps.co.id.git tps-app

Perintah ini mengunduh seluruh repository ke folder /opt/tps-app.

3 Verifikasi Git Repository

Ketik perintah berikut untuk memastikan repository berhasil di-clone:

cd /opt/tps-app
git status
git log --oneline -5

Perintah pertama menunjukkan branch aktif (seharusnya main). Perintah kedua menunjukkan 5 commit terakhir.

3.3 Perubahan Docker Compose

Perubahan utama pada docker-compose.yml adalah pada bagian volumes. Volume mount yang sebelumnya mengarah ke path /opt/tps-app/pkgs (memerlukan symlink) kini langsung mengarah ke path di dalam Git repository: /opt/tps-app/tps-core/tps/pkgs. Dengan pendekatan ini, tidak diperlukan symlink dan proses setup/recovery server menjadi lebih sederhana.

Docker Compose Terbaru

4 Update File docker-compose.yml

Buka file docker-compose.yml dengan editor:

nano /opt/tps-app/docker-compose.yml

Ganti seluruh isinya dengan konfigurasi berikut:

services:
  tps-app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: tps-app
    restart: unless-stopped
    ports:
      - "3300:3300"
    environment:
      - NODE_ENV=production
      - DATABASE_URL=postgresql://postgres:TpsDB2026Secure@172.19.154.92:5432/tps_web
      - PORT=3300
    volumes:
      # Source code (langsung dari path Git repository)
      - /opt/tps-app/tps-core/tps/pkgs:/app/pkgs
      - /opt/tps-app/tps-core/tps/app/srv:/app/app/srv
      - /opt/tps-app/tps-core/tps/app/web/deploy:/app/app/web/deploy
      # Database config (terpisah dari Git, berisi .env)
      - /opt/tps-app/app/db:/app/app/db
      # Data (file upload dan logs)
      - /opt/tps-data/uploads:/data/files
      - /opt/tps-data/logs:/app/logs
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3300/"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s

Simpan file (Ctrl+O, Enter) dan keluar (Ctrl+X).

Mengapa volume mount langsung ke path Git?
  • Tidak perlu symlink — mengurangi kompleksitas dan potensi error.
  • Recovery cepat — setelah git clone, Docker Compose langsung bisa dijalankan tanpa konfigurasi tambahan.
  • Git pull langsung berlaku — perubahan dari git pull otomatis terlihat oleh container karena Docker langsung membaca file dari path Git.

3.4 Konfigurasi Database (app/db)

Direktori /opt/tps-app/app/db/ merupakan satu-satunya direktori yang tidak diambil dari Git repository. Direktori ini berisi konfigurasi database yang bersifat rahasia dan spesifik untuk lingkungan server.

PENTING: Direktori app/db berisi file .env dengan kredensial database yang tidak boleh disimpan di Git. Direktori ini harus dibuat manual dan di-backup secara terpisah.
5 Buat Direktori dan File Database Config

Ketik perintah berikut satu per satu:

# Buat direktori
mkdir -p /opt/tps-app/app/db/prisma

# Buat file .env (kredensial database)
cat > /opt/tps-app/app/db/.env << 'EOF'
DATABASE_URL="postgresql://postgres:TpsDB2026Secure@172.19.154.92:5432/tps_web"
EOF

# Buat file db.ts (export PrismaClient)
cat > /opt/tps-app/app/db/db.ts << 'EOF'
export { PrismaClient } from "@prisma/client";
EOF

# Copy schema Prisma dari Git repository
cp /opt/tps-app/tps-core/tps/app/db/prisma/schema.prisma \
   /opt/tps-app/app/db/prisma/
6 Generate Prisma Client di dalam Container

Setelah container berjalan, generate Prisma client:

docker exec tps-app bash -c "cd /app/app/db && bunx prisma generate --schema=prisma/schema.prisma"

3.5 Prosedur Deployment Baru (via Git Pull)

RINGKASAN: Deployment kini hanya memerlukan 2 perintah yang dijalankan di server: 
cd /opt/tps-app && git pull origin main && docker restart tps-app

Prosedur Lengkap (Langkah demi Langkah)

1 Login ke Server

Buka AnyDesk dan hubungkan ke server, lalu buka terminal. Atau gunakan SSH:

ssh root@172.19.154.93
2 Masuk ke Direktori Aplikasi dan Tarik Perubahan Terbaru

Ketik perintah berikut lalu tekan Enter:

cd /opt/tps-app
git pull origin main

Output yang normal:

remote: Enumerating objects: 15, done.
remote: Counting objects: 100% (15/15), done.
Updating abc1234..def5678
Fast-forward
 tps-core/tps/app/srv/api/_frontend.ts | 5 ++---
 tps-core/tps/pkgs/server/create.ts    | 2 +-
 2 files changed, 3 insertions(+), 4 deletions(-)

Jika muncul Already up to date., artinya tidak ada perubahan baru. Tidak perlu restart container.

3 Restart Container

Ketik perintah berikut lalu tekan Enter:

docker restart tps-app

Tunggu sekitar 15–20 detik agar container selesai startup.

4 Verifikasi

Ketik perintah berikut untuk memastikan container berjalan normal:

# Cek status container
docker ps --filter name=tps-app

# Cek homepage
curl -s -o /dev/null -w "%{http_code}" http://localhost:3300/

Container harus berstatus Up dan homepage mengembalikan kode 200.

3.6 Prosedur Recovery Server (dari Nol)

Salah satu keuntungan utama dari pendekatan tanpa symlink adalah proses recovery yang sangat cepat. Jika server perlu di-setup ulang dari nol (misalnya setelah reinstall OS atau migrasi server), seluruh proses dapat dilakukan dalam 4 langkah utama.

RECOVERY HANYA 4 LANGKAH:
  1. Install Docker
  2. Clone Git repository
  3. Buat konfigurasi database (app/db)
  4. Jalankan docker compose up

Estimasi waktu: 10–15 menit (dengan koneksi internet).

Langkah Recovery Lengkap

1 Install Docker Engine 
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh
sudo systemctl enable docker
sudo systemctl start docker
2 Clone Repository dan Buat Direktori Data 
cd /opt
git clone https://github.com/agussang/tps.co.id.git tps-app
mkdir -p /opt/tps-data/uploads /opt/tps-data/logs
3 Buat Konfigurasi Database 
mkdir -p /opt/tps-app/app/db/prisma

cat > /opt/tps-app/app/db/.env << 'EOF'
DATABASE_URL="postgresql://postgres:TpsDB2026Secure@172.19.154.92:5432/tps_web"
EOF

cat > /opt/tps-app/app/db/db.ts << 'EOF'
export { PrismaClient } from "@prisma/client";
EOF

cp /opt/tps-app/tps-core/tps/app/db/prisma/schema.prisma \
   /opt/tps-app/app/db/prisma/
4 Build dan Jalankan Container 
cd /opt/tps-app
docker compose build
docker compose up -d

Tunggu 40 detik untuk startup, lalu verifikasi:

docker ps --filter name=tps-app
curl -s -o /dev/null -w "%{http_code}" http://localhost:3300/
CATATAN RECOVERY:
  • File upload (9.3 GB) harus di-restore terpisah ke /opt/tps-data/uploads/ dari backup.
  • Database berada di server terpisah (172.19.154.92) sehingga tidak terpengaruh oleh recovery application server.
  • Jika Prisma perlu di-generate ulang: docker exec tps-app bash -c "cd /app/app/db && bunx prisma generate --schema=prisma/schema.prisma"

4.2 Update Dependencies (Berkala)

Prosedur ini dilakukan jika ada perubahan pada file package.json (penambahan atau update library). Prosedur ini memerlukan rebuild Docker image karena node_modules disimpan di dalam image, bukan di volume mount. 

# Langkah lengkap
cd /opt/tps-app
git pull origin main
docker compose build --no-cache
docker compose up -d
PENTING: Proses docker compose build memerlukan akses internet dan memakan waktu 5–15 menit. Pastikan koneksi internet tersedia sebelum menjalankan perintah ini.

4.3 Rebuild Docker Image

Rebuild Docker image diperlukan dalam situasi berikut:

  • Ada perubahan pada package.json (dependencies baru/update).
  • Ada perubahan pada Dockerfile.
  • Perlu update base image Bun (security patch).
# Rebuild dan restart (satu perintah)
cd /opt/tps-app && docker compose build --no-cache && docker compose up -d

4.4 Backup Database

Backup database disarankan dilakukan secara berkala (minimal mingguan) untuk keperluan disaster recovery. Database berada di server terpisah (172.19.154.92), sehingga tidak terpengaruh oleh masalah pada application server. 

# Backup database (jalankan di server 172.19.154.93)
PGPASSWORD=TpsDB2026Secure pg_dump \
  -h 172.19.154.92 -U postgres -d tps_web \
  --no-owner --no-acl \
  > /opt/tps-data/backup_$(date +%Y%m%d_%H%M%S).sql
Catatan: File backup disimpan di /opt/tps-data/. Pastikan direktori ini di-backup ke lokasi terpisah (external storage atau cloud) secara berkala.

Lampiran D — Prosedur Recovery dari Nol (Disaster Recovery)

Prosedur ini digunakan jika server perlu di-setup ulang sepenuhnya (misalnya setelah reinstall OS atau migrasi ke server baru).

PRASYARAT:
  • Server memiliki akses internet
  • Database server (172.19.154.92) sudah berjalan dan dapat diakses
  • Backup file upload tersedia (untuk restore ke /opt/tps-data/uploads/)
# ============================================
# PROSEDUR RECOVERY DARI NOL
# Estimasi waktu: 10-15 menit
# ============================================

# LANGKAH 1: Install Docker
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh
sudo systemctl enable docker && sudo systemctl start docker

# LANGKAH 2: Clone Repository
cd /opt
git clone https://github.com/agussang/tps.co.id.git tps-app
mkdir -p /opt/tps-data/uploads /opt/tps-data/logs

# LANGKAH 3: Buat Konfigurasi Database
mkdir -p /opt/tps-app/app/db/prisma

cat > /opt/tps-app/app/db/.env << 'EOF'
DATABASE_URL="postgresql://postgres:TpsDB2026Secure@172.19.154.92:5432/tps_web"
EOF

cat > /opt/tps-app/app/db/db.ts << 'EOF'
export { PrismaClient } from "@prisma/client";
EOF

cp /opt/tps-app/tps-core/tps/app/db/prisma/schema.prisma \
   /opt/tps-app/app/db/prisma/

# LANGKAH 4: Build dan Jalankan
cd /opt/tps-app
docker compose build
docker compose up -d

# LANGKAH 5: Verifikasi (setelah 40 detik)
sleep 40
docker ps --filter name=tps-app
curl -s -o /dev/null -w "%{http_code}" http://localhost:3300/

# LANGKAH 6 (opsional): Restore file upload dari backup
# scp backup-server:/path/to/files_backup.tar.gz /opt/tps-data/
